> 단순한 미들웨어 base class 하나를 교체해 LiteLLM proxy의 지연 시간과 처리량을 개선한 과정입니다.

---

## 기존 구성

LiteLLM proxy server에는 두 개의 middleware layer가 있습니다. 첫 번째는 FastAPI가 다시 export하는 Starlette의 `CORSMiddleware`이며 pure ASGI middleware입니다. 그다음에는 `PrometheusAuthMiddleware`라는 단순한 `BaseHTTPMiddleware`가 있습니다.

`PrometheusAuthMiddleware`의 역할은 `/metrics` endpoint 요청을 인증하는 것입니다. 기본으로 켜져 있지는 않고, proxy config의 flag로 활성화합니다.

<details>
<summary>Proxy config flag 설정</summary>

```yaml
litellm_settings:
    require_auth_for_metrics_endpoint: true
```

</details>

이 middleware는 두 가지를 확인합니다. 요청이 `/metrics`를 향하는지, 그리고 인증이 활성화되어 있는지입니다. 대부분의 요청에서는 두 조건이 모두 성립하지 않으므로 요청을 그대로 통과시킵니다.

<details>
<summary>PrometheusAuthMiddleware 소스</summary>

```python
class PrometheusAuthMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request: Request, call_next):
        if self._is_prometheus_metrics_endpoint(request):
            if self._should_run_auth_on_metrics_endpoint() is True:
                try:
                    await user_api_key_auth(request=request, api_key=...)
                except Exception as e:
                    return JSONResponse(status_code=401, content=...)
        response = await call_next(request)
        return response

    @staticmethod
    def _is_prometheus_metrics_endpoint(request: Request):
        if "/metrics" in request.url.path:
            return True
        return False
```

</details>

겉으로는 문제없어 보입니다. `BaseHTTPMiddleware`를 subclass하고 `dispatch()`를 구현하면 끝입니다. Starlette 문서<sup>[1](#footnote-1)</sup>에서도 이런 패턴을 볼 수 있습니다.

## BaseHTTPMiddleware가 실제로 하는 일

`dispatch()` method를 작성하면 요청이 함수 안으로 들어와 그대로 반대편으로 빠져나갈 것처럼 보입니다. 하지만 실제 동작은 훨씬 더 복잡합니다.

아무 작업도 하지 않는 pure passthrough 요청에서도 `BaseHTTPMiddleware`는 매번 **7개의 중간 객체와 task**를 만듭니다.

<BaseHTTPMiddlewareAnimation />

body state를 추적하기 위해 요청을 새 객체로 감싸고, synchronization event를 만들고, middleware와 내부 app 사이에서 message를 전달할 in-memory channel을 할당합니다. lifecycle 관리를 위한 task group도 만들고, `call_next()`를 호출하면 실제 route handler를 *별도 background task*에서 실행합니다. 이후 response body는 다시 그 in-memory channel을 거쳐 streaming response object로 감싸진 뒤 호출자에게 도달합니다. 단순 passthrough에는 과한 비용입니다.

우리 경우처럼 99.9%의 요청에서 아무 일도 하지 않는 middleware라면 이 비용을 매번 지불할 이유가 없습니다.

반면 pure ASGI middleware는 request path만 확인하고 바로 다음 단계로 넘길 수 있습니다.

<PureASGIAnimation />

이 middleware가 하는 일은 매우 단순합니다. 대부분의 요청에서는 아무 것도 하지 않고 그대로 통과시키면 됩니다. task group, memory stream, cancel scope가 필요한 일이 아니라 함수 호출 하나면 충분합니다.

## 두 방식 비교

`BaseHTTPMiddleware` subclass를 pure ASGI middleware로 교체했습니다. 차이를 측정하기 위해 Apache Bench<sup>[2](#footnote-2)</sup>로 LiteLLM middleware stack의 두 구성을 비교했습니다. 기존 구성은 pure ASGI 1개와 `BaseHTTPMiddleware` 1개였고, 새 구성은 pure ASGI 2개입니다.

최소 FastAPI app은 `GET /health` 요청에 `PlainTextResponse("ok")`만 반환합니다. middleware overhead만 분리해서 보기 위해 endpoint는 아무 작업도 하지 않습니다. 구성 간 차이는 middleware plumbing 자체의 비용입니다. 두 middleware 모두 다음 layer를 호출할 뿐입니다. 하는 일은 같고 base class만 다릅니다.

Apache Bench(`ab`)는 단일 uvicorn worker와 1,000개의 동시 connection으로 서버에 요청을 보냅니다. worker 하나는 event loop 하나를 의미하므로, 이 benchmark는 각 middleware 설계가 단일 thread에서 동시 부하를 어떻게 처리하는지 직접 측정합니다.

<BenchmarkVisualization />

<details>
<summary>직접 실행해 보기</summary>

아래 script를 `benchmark_middleware.py`로 저장한 뒤 실행합니다.

```bash
# Terminal 1 — start the "before" server (1 ASGI + 1 BaseHTTPMiddleware)
python benchmark_middleware.py --middleware mixed

# Terminal 2 — benchmark it
ab -n 50000 -c 1000 http://localhost:8000/health

# Stop the server, then start the "after" server (2x pure ASGI)
python benchmark_middleware.py --middleware asgi

# Terminal 2 — benchmark again
ab -n 50000 -c 1000 http://localhost:8000/health
```

```python
import argparse
import uvicorn
from fastapi import FastAPI
from fastapi.responses import PlainTextResponse
from starlette.middleware.base import BaseHTTPMiddleware
from starlette.requests import Request
from starlette.types import ASGIApp, Receive, Scope, Send


class NoOpBaseHTTPMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request: Request, call_next):
        return await call_next(request)


class NoOpPureASGIMiddleware:
    def __init__(self, app: ASGIApp) -> None:
        self.app = app

    async def __call__(self, scope: Scope, receive: Receive, send: Send) -> None:
        await self.app(scope, receive, send)


def create_app(middleware_type: str | None = None, layers: int = 2) -> FastAPI:
    app = FastAPI()

    @app.get("/health")
    async def health():
        return PlainTextResponse("ok")

    if middleware_type == "mixed":
        app.add_middleware(NoOpBaseHTTPMiddleware)
        app.add_middleware(NoOpPureASGIMiddleware)
    elif middleware_type == "asgi":
        for _ in range(layers):
            app.add_middleware(NoOpPureASGIMiddleware)

    return app


if __name__ == "__main__":
    parser = argparse.ArgumentParser()
    parser.add_argument("--middleware", choices=["asgi", "mixed"], default=None)
    parser.add_argument("--layers", type=int, default=2)
    parser.add_argument("--port", type=int, default=8000)
    args = parser.parse_args()

    app = create_app(middleware_type=args.middleware, layers=args.layers)
    uvicorn.run(app, host="0.0.0.0", port=args.port, workers=1, log_level="warning")
```

## 변경 내용

교체한 구현은 다음과 같습니다.

```python
class PrometheusAuthMiddleware:
    def __init__(self, app: ASGIApp) -> None:
        self.app = app

    async def __call__(self, scope: Scope, receive: Receive, send: Send) -> None:
        if scope["type"] != "http" or "/metrics" not in scope.get("path", ""):
            await self.app(scope, receive, send)
            return

        if litellm.require_auth_for_metrics_endpoint is True:
            request = Request(scope, receive)
            api_key = request.headers.get("Authorization") or ""
            try:
                await user_api_key_auth(request=request, api_key=api_key)
            except Exception as e:
                # send 401 directly via ASGI protocol
                ...
                return

        await self.app(scope, receive, send)
```

`/metrics`를 향하지 않는 99.9%의 요청에서 middleware 작업은 이제 dict lookup 하나, string check 하나, function call 하나로 끝납니다. 객체를 할당하지 않고 task도 만들지 않습니다.

소프트웨어가 커지고 더 많은 책임을 맡게 될수록 지금 쓰는 도구가 그 작업에 맞는지 계속 평가해야 합니다. 새로 추가되는 middleware에서 같은 문제가 반복되지 않도록 static analysis check도 넣고 있습니다. 필요한 use case가 확인되면 다시 평가하겠지만, 현재 LiteLLM이 해야 하는 작업에는 이 비용이 필요하지 않았습니다.

이번 middleware 변경은 LiteLLM proxy 최적화 작업의 한 부분입니다. 지난 2주 동안의 전체 최적화를 합산하면 proxy overhead가 약 **30% 감소**한 것으로 측정했습니다.
